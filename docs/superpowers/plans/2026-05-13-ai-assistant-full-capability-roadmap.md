# AI 助手完整能力阶段路线图

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement each approved phase task-by-task. This document is a product and engineering roadmap, not a single-session implementation checklist.

**Goal:** 将 AI 助手逐步建设为会议室管理系统的全系统自然语言操作入口。

**Architecture:** 以 Planner + Tool Registry 为主干。Ollama 负责自然语言理解和结构化计划生成，后端工具负责权限校验、参数校验、业务执行、确认流程和降级兜底。RAG 只用于知识问答和帮助内容增强，不作为业务操作执行主路径。

**Tech Stack:** Vue 3 + Vite + TypeScript + Element Plus；Spring Boot + Spring AI + Ollama + MyBatis；现有业务 service 和 controller 能力。

---

## 总体原则

- 每个阶段都必须可独立验收，不能把用户体验押到最后一次大重构。
- 读操作可以直接执行并返回结果卡片。
- 写操作必须先返回确认卡片，用户确认后执行冻结参数快照。
- 模型失败不能导致助手整体不可用；必须降级到确定性解析、追问或可恢复错误卡片。
- 普通用户只能调用普通端工具；管理员工具必须校验角色。
- 后续新增工具必须先进入 Tool Registry，再接入 Planner 和前端卡片。

## 阶段 0：稳定性止血与链路统一

**目标：** 在第一阶段重构未完全完成前，避免继续出现“消息服务暂时不可用”这类不可接受体验。

**范围：**
- 明确 AI 助手页面只走一个主入口。
- 前端不再把普通接口异常直接展示成“当前消息服务暂时不可用”。
- 后端对 Ollama 超时、未启动、非法 JSON 做可恢复响应。
- 对“今天有哪些会”“明天有哪些会”“明天有什么安排”做最低限度回归保护。

**验收：**
- 关闭 Ollama 后，用户问“明天有哪些会”不会出现消息服务不可用。
- 接口异常时页面展示可恢复提示或追问。

**非目标：**
- 不新增全系统工具。
- 不改完整 UI 卡片体系。

## 阶段 1：Planner 核心与现有助手动作全集

**目标：** 完成当前规格文档定义的第一阶段，让 AI 助手具备稳定的 Planner、Tool Registry、确认执行和卡片交互基础。

**参考规格：**
- `docs/superpowers/specs/2026-05-13-ai-assistant-planner-design.md`

**范围：**
- 概览查询。
- 今日安排查询。
- 日历查询。
- 会议室查询。
- 会议室详情。
- 我的预约列表。
- 预约详情。
- 创建预约。
- 修改预约。
- 取消预约。
- 预约评价。
- 管理员待审核预约查询。
- 管理员通过预约。
- 管理员驳回预约。

**验收：**
- “今天有哪些会”“明天有哪些会”“我明天有会吗”“明天有什么安排”“明天有哪些预约”稳定返回结果或合理追问。
- 所有写操作都有确认卡片。
- 普通用户不能执行管理员工具。
- Ollama 不可用时读操作仍能兜底完成或追问。
- 前端可渲染文本、查询结果、参数表单、确认、执行结果、错误卡片。

**非目标：**
- 不覆盖会议室管理、设备管理、通知发布、统计深度分析等新增管理员工具。

## 阶段 2：管理员资源管理工具扩展

**目标：** 让管理员能通过 AI 助手完成后台资源管理的核心 CRUD 和状态变更。

**范围：**
- 会议室管理：
  - 查询会议室列表。
  - 查看会议室详情。
  - 新增会议室。
  - 修改会议室基础信息。
  - 启用、停用或维护会议室。
- 设备管理：
  - 查询设备列表。
  - 新增设备。
  - 修改设备信息。
  - 启用、停用设备。
- 会议室设备绑定：
  - 查询某会议室已绑定设备。
  - 绑定设备到会议室。
  - 解绑设备。
  - 查看设备绑定统计。

**交互要求：**
- 查询类返回表格或摘要卡片。
- 新增、修改、启停、绑定、解绑全部走确认卡片。
- 对名称相近的会议室或设备必须返回候选卡片，不能猜测执行。

**验收：**
- 管理员可以说“把远景会议室设为维护中”，助手返回确认卡片，确认后执行。
- 管理员可以说“给 A101 绑定投影仪”，助手能在设备歧义时要求选择。
- 普通用户请求这些工具时返回权限不足卡片。

**非目标：**
- 不做批量导入。
- 不做跨系统资产同步。

## 阶段 3：通知、统计与运营分析工具

**目标：** 让管理员能用自然语言完成通知发布和常用运营分析。

**范围：**
- 通知发布：
  - 创建系统通知。
  - 指定通知类型、标题、内容、跳转链接。
  - 发布前确认。
- 通知查询：
  - 查看最近通知。
  - 查看未读或重要通知。
- 统计分析：
  - 查询会议室使用率。
  - 查询预约趋势。
  - 查询热门会议室。
  - 查询设备绑定和使用概况。
  - 查询待处理预约数量。

**交互要求：**
- 统计结果优先使用图表摘要卡片或指标卡片。
- 通知发布必须确认。
- AI 可以解释指标含义，但数字必须来自后端接口。

**验收：**
- 管理员可以说“发布一条明天上午系统升级通知”，助手追问缺失字段或生成确认卡片。
- 管理员可以说“本周会议室使用率怎么样”，助手返回统计卡片。
- 统计回答中的数字可追溯到后端接口返回。

**非目标：**
- 不生成复杂商业分析报告。
- 不引入外部 BI 系统。

## 阶段 4：系统知识与 RAG 帮助中心

**目标：** 让 AI 助手能回答系统规则、操作帮助、字段解释和会议室使用说明。

**范围：**
- 建立知识来源：
  - 用户操作说明。
  - 管理员操作说明。
  - 预约规则。
  - 审批规则。
  - 会议室和设备说明。
  - 常见问题。
- 建立轻量检索能力：
  - 按关键词、主题、页面、角色检索。
  - 回答时引用知识条目标题。
- 将 RAG 与工具调用分流：
  - 操作请求进入 Planner。
  - 规则和帮助问题进入知识问答。
  - 混合问题先回答规则，再引导执行工具。

**验收：**
- 用户问“怎么取消预约”，助手能说明规则并提供“帮我取消预约”的操作建议。
- 管理员问“审批驳回需要填原因吗”，助手能基于知识条目回答。
- 用户问“帮我取消明天下午的会议，取消规则是什么”，助手先说明规则，再进入取消确认流程。

**非目标：**
- RAG 不替代业务接口查询。
- 不允许 RAG 内容绕过权限。

## 阶段 5：多轮任务编排与跨工具流程

**目标：** 支持一个自然语言目标拆成多个工具步骤，让助手能完成复杂工作流。

**范围：**
- 多轮上下文：
  - 记住最近查询结果。
  - 支持“就这个会议室”“把刚才那个会改到明天”。
- 跨工具编排：
  - 查空闲会议室后创建预约。
  - 查预约冲突后推荐改期。
  - 查询待审预约后逐条审批。
  - 发布通知后跳转查看通知列表。
- 任务状态：
  - 展示当前任务进度。
  - 支持取消当前任务。
  - 支持回退到上一步补充参数。

**验收：**
- 用户说“帮我找明天下午 10 人会议室，然后预约项目周会”，助手先查会议室，再让用户选择，最后确认创建。
- 用户说“把刚才那个会改到后天下午”，助手能基于上下文定位候选预约并确认修改。
- 任一步不确定时不会猜测执行。

**非目标：**
- 不做自动后台长期任务。
- 不做无需用户确认的批量写操作。

## 阶段 6：质量、观测与可运营化

**目标：** 让 AI 助手从“能用”变成“可维护、可评估、可持续优化”。

**范围：**
- 日志与审计：
  - 记录 toolName、参数摘要、用户角色、确认结果、执行结果。
  - 写操作保留执行审计。
- 质量评估：
  - 建立自然语言回归用例集。
  - 覆盖高频同义说法。
  - 记录模型解析失败率、fallback 命中率、追问率。
- 管理配置：
  - 可开关 Ollama 解析。
  - 可配置模型超时。
  - 可查看工具注册完整性。
- 安全治理：
  - 敏感操作二次确认。
  - 权限失败可追踪。
  - 防止 prompt 注入影响工具执行。

**验收：**
- 每次确认执行都有审计记录。
- 可以用固定测试集验证助手理解能力是否退化。
- 模型不可用时的 fallback 命中情况可被统计。

**非目标：**
- 不做跨租户能力。
- 不做用户行为画像推荐。

## 阶段 7：完整全系统操作入口

**目标：** 达到产品层面的完整能力：用户可以在权限范围内通过 AI 助手发现、查询、创建、修改、审批、分析系统主要对象。

**能力边界：**
- 普通用户：
  - 查概览。
  - 查会议室。
  - 查日历。
  - 查、建、改、取消、评价预约。
  - 查通知。
  - 查询操作帮助和规则。
- 管理员：
  - 普通用户全部能力。
  - 审批预约。
  - 管理会议室。
  - 管理设备。
  - 管理会议室设备绑定。
  - 发布通知。
  - 查看运营统计。
  - 查询系统规则和管理帮助。

**最终验收：**
- 每个前端主页面的核心功能至少有一个对应 AI 工具。
- 每个写工具都有确认流程、权限校验、审计记录。
- 用户自然语言表达不依赖单一关键词。
- 模型失败时系统可继续服务核心确定性任务。
- 前端卡片能承载所有工具结果和确认流程。

## 阶段推进建议

| 阶段 | 优先级 | 依赖 | 建议验收方式 |
| --- | --- | --- | --- |
| 阶段 0 | 最高 | 无 | 手动回归 + 最小测试 |
| 阶段 1 | 最高 | 阶段 0 | 后端单测 + 前端单测 + 浏览器流程 |
| 阶段 2 | 高 | 阶段 1 | 管理员工具测试 + 权限测试 |
| 阶段 3 | 高 | 阶段 1 | 通知发布流程 + 统计卡片测试 |
| 阶段 4 | 中 | 阶段 1 | 知识问答回归集 |
| 阶段 5 | 中 | 阶段 2、3 | 多轮任务场景测试 |
| 阶段 6 | 中 | 阶段 1 | 审计、指标、回归集 |
| 阶段 7 | 最终整合 | 阶段 1-6 | 全系统端到端验收 |

## 执行注意

- 当前全栈 agent 正在执行阶段 1 时，其他 agent 不应同时改同一批 AI 助手核心文件。
- 后续阶段应在阶段 1 合并稳定后再拆分执行。
- 每个阶段开始前都应把该阶段拆成独立实施计划。
- 如果发现某阶段需要大幅改动业务 service，应先单独评估，不要在 AI 助手任务里顺手重构业务层。
