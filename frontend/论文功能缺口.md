## 1. 管理端预约审核、驳回、异常处理

状态：已补齐

论文依据：
- 管理员应能查看预约申请。
- 管理员应能审核、驳回预约。
- 管理员应能处理异常预约，并保留处理记录。

当前现状：
- 前端已新增管理端预约审核页面，路径为 `/admin/reservations`。
- 前端已接入管理端预约列表、通过、驳回、异常处理接口约定。
- 后端已新增管理端预约列表、审核通过、驳回、异常处理接口。
- 普通用户创建预约默认进入 `PENDING`，不再直接生效。
- `PENDING` 不计入正式会议室占用；审核通过时才转为 `ACTIVE`。
- 审核通过时会重新校验会议室状态、容量、时间冲突、设备需求。
- 审核操作已记录处理人、处理时间、处理备注或原因，并生成通知。

已实现接口：
- `GET /api/v1/admin/reservations`
- `PATCH /api/v1/admin/reservations/{id}/approve`
- `PATCH /api/v1/admin/reservations/{id}/reject`
- `PATCH /api/v1/admin/reservations/{id}/exception`

后端实现记录：
- 入口：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\controller\AdminReservationController.java`
- 服务：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\service\impl\ReservationServiceImpl.java`

- 数据库字段：`approval_remark`、`reject_reason`、`exception_reason`、`processed_by`、`processed_at`

验收标准：
- 管理员可以在后台看到待处理、已通过、已驳回、异常等预约。
- 管理员可以完成通过、驳回、异常处理操作。
- 普通用户无法进入管理端预约页面。
- 操作后预约状态正确变化，用户侧预约列表同步展示新状态。
- 操作后用户可以收到对应通知。

## 4. 登录安全策略增强

状态：已补齐

论文依据：
- 登录需要验证码。
- 密码应进行摘要校验。
- 连续错误登录应限制并临时锁定。
- 会话应有过期控制。

当前现状：
- 验证码已实现。
- 登录密码已改为 SHA-256 摘要校验，不再接受明文 `password_hash`。
- 连续登录失败会记录用户名维度的失败次数，达到阈值后临时锁定。
- `TokenStore` 已为 token 设置过期时间，过期后返回未登录。
- 前端 axios 已对 401 执行退出登录和刷新。

后端实现记录：
- 密码摘要：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\auth\PasswordHasher.java`
- 登录失败锁定：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\auth\LoginAttemptGuard.java`
- token 过期：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\auth\TokenStore.java`
- 登录服务：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\java\com\llf\service\impl\AuthServiceImpl.java`
- 数据迁移脚本：`E:\project\javaProject\meeting_room\meeting-room-server\src\main\resources\sql\password_security.sql`

验收标准：
- 密码不再明文比较。
- 连续输错达到阈值后短时间内不能继续登录。
- token 过期后接口返回 401。
- 正常登录流程不受影响。

验证结果：
- 后端目标测试通过：`AuthServiceImplTest`、`TokenStoreTest`、`AuthInterceptorTest` 共 9 个用例通过。
