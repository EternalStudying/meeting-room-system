## 关键问题
1. 前端是否彻底停止调用旧 `/api/v1/ai/chat` 作为助手主页面入口？
2. 卡片样式应尽量复用当前页面视觉，还是允许整体重做助手页布局？

## 已做决策
| 决策 | 理由 |
|------|------|
| 确认前只调研和计划，不改业务代码 | 用户明确要求先输出计划并等待确认 |
| 前端记录独立放在 `frontend/` 下 | 用户要求前后端目录分别记录各自更改 |
| 不新增前端依赖 | Vue、Pinia、Element Plus、Vitest 已足够实现 cards UI |
| 助手页不再使用旧 stage 协议作为主渲染依据 | 新协议以 `state/cards` 驱动卡片，避免旧异常文案和旧面板分支继续影响主流程 |
| 接口异常落为本地 error card | 前端展示可恢复提示，不再显示“当前消息服务暂时不可用” |
| 管理端会议室修复先写回归测试 | 4 个问题均为可回归行为，先证明现有缺陷再改最小实现 |
| 新增会议室设备选择复用设备库存数据 | 避免新增依赖和额外接口，创建成功后调用现有绑定接口完成设备归属 |
| 管理端会议室列表数据进入页面状态前先归一化 | 后端自建会议室可返回 `description: null` 和历史数字状态，前端详情/编辑应消费稳定字段 |
| 删除会议室使用页面内 `el-dialog`，不再使用 `ElMessageBox.confirm` | 管理页已有统一弹窗视觉；页面内弹窗可避免 MessageBox 在该布局下位置异常 |
| 删除会议室接口启用本地错误处理 | 避免 axios 全局提示和页面 catch 同时提示；页面可以展示“已有预约记录”的明确原因 |
| 设备新增/编辑/删除接口启用本地错误处理 | 避免 axios 全局提示和页面 catch 同时提示；设备页可以展示重复编码等业务原因 |
| 删除设备使用页面内 `el-dialog` | 与管理端其它弹窗统一，避免 MessageBox 在设备页左上角且内容不可见 |
| 登录验证码错误在 axios 全局归一化处理 | 后端返回 `captcha invalid`，登录页复用统一 request 错误处理；只补映射能最小改动修正文案 |
| 助手欢迎语重复显示在渲染层过滤同文案 text card | 后端 cards 协议仍可保留兼容字段；前端只避免同一个 turn 的主消息和纯文本卡片重复展示 |
| 助手页面只渲染需要用户操作的卡片 | 查询、错误和执行结果信息已在聊天气泡中展示；确认卡片保留以满足写操作必须确认的安全要求 |
| `clarification` 属于需要用户继续补充的交互卡片 | Planner v2 的“取消这个会议室”必须追问，前端需要显示澄清卡片 |
| RAG `text` 回答仍不渲染额外卡片 | 系统知识回答不需要用户补参或确认，聊天气泡已足够 |
| 补参卡片只渲染当前活动 turn | 旧补参卡片继续绑定全局 `activeFieldValues`，提交后会显示已选参会人被清空；隐藏旧卡片更符合聊天流状态 |
| 创建预约的设备需求不放在首张基础补参卡 | 先收主题/日期/时间/参会人，再由后端返回设备需求补参卡，之后按设备筛选推荐会议室 |
| 紧急会议入口仅放在管理端页面 | 规格要求普通用户不可见入口，用户端预约和日历页面不加抢占入口 |
| 预约日历页不再展示紧急会议入口 | 用户要求管理员登录预约日历页删除“添加/新建紧急会议”按钮；紧急会议创建入口保留在管理端预约审核页 |
| 通知发布入口收敛到顶部铃铛弹层 | 通知发布是低频动作，侧边栏独立页面移除；admin 在铃铛弹层发布，普通用户不显示入口 |
| 预约创建弹窗双栏使用 grid stretch + panel flex 拉伸 | grid item 会拉伸，但内部 `.panel-shell` 需要通过父级 flex 承接高度，才能让“预约信息”和“智能推荐”外框等高 |
| 智能推荐列表参与纵向 flex 分配 | 外框等高还不够；推荐列表如果保留固定 `max-height`，按钮会停在半截，视觉上仍不对齐 |
| 普通预约和紧急会议使用不同高度策略 | 普通预约左侧短，应缩右侧；紧急会议左侧长，应拉右侧。单一 `align-items: stretch` 会让普通预约左侧产生大块空白 |
| 本次整理只删除无引用前端遗留文件 | 后端旧 AI Chat 链路仍有公开接口和测试覆盖，不能视为无用代码直接删除 |
| 保留 `frontend/LICENSE` 原模板版权声明 | 项目基于模板演进，删除版权声明不合规；模板作者信息只从项目元信息和无意义示例测试中清理 |
