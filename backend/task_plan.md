## 关键问题
1. 第一阶段是否保留旧 `/api/v1/ai/chat`，但让前端助手页只调用 `/api/v1/ai/assistant`？
2. 新版 turn DTO 是否可以短期兼容旧字段以降低前后端同步风险？

## 已做决策
| 决策 | 理由 |
|------|------|
| 确认前只调研和计划，不改业务代码 | 用户明确要求先输出计划并等待确认 |
| 后端工具 executor 复用现有业务 service | 规格要求工具负责真实业务执行，模型不得直接操作数据库 |
| 新增 Tool Registry 时保留现有业务 handler 逻辑作为 executor 来源 | 第一阶段要避免无关重构，当前 handler 已覆盖大部分业务执行 |
| 不新增后端依赖 | 现有 Spring Boot、Jackson、Spring AI、JUnit/Mockito 已足够 |
| 旧 `/api/v1/ai/chat` 本阶段保持隔离不删除 | 助手主页面已走 `/api/v1/ai/assistant`，旧入口可能仍被其他调用方依赖，删除会扩大风险 |
| `AiAssistantTurnVO` 暂时兼容旧字段 | 前后端已使用新版 `state/cards`，保留旧字段降低已有测试和外部调用破坏面 |
| 选中预约不可用时返回错误卡片而不是重新追问目标预约 | 用户已经完成选择，重复追问会形成死循环；错误卡片更符合可恢复提示 |
| 取消/修改/评价在确认前过滤不可操作预约 | 写操作必须先确认，但确认前也必须确保用户有权且当前状态允许执行 |
| AI 助手无关语句返回能力范围提示而非继续追问 | 非业务语句没有可补参数，继续追问会误导用户进入任务流程 |
| “我的会议/本周预约”类表达优先走 `reservations.list` | 避免 Ollama 或通用日历/创建规则导致同义表达返回不一致 |
| 预约查询列表仅展示 ACTIVE/ENDED | CANCELLED、REJECTED、EXCEPTION、PENDING 都不是用户需要参加的有效会议 |
| Planner v2 主路径改为 LLM-first，规则只在 Planner 失败时 fallback | 满足 v2 规格要求，同时保留 Ollama 不可用时的稳定查询能力 |
| RAG 只回答系统规则和操作说明，不生成业务执行 plan | 避免知识问答绕过 Tool Registry 或写操作确认 |
| `取消这个会议室` 永远追问 | “会议室”不是可取消实体，不能把它隐式映射为取消预约 |
| LLM 字段值不可信时要清理或 fallback | 实测 Ollama 可能输出“下周开始日期”“明天”等自然语言字段，不能直接传给业务 service |
| 自然语言里的数字不能默认等同于本人预约 ID | 用户界面展示的是会议主题/会议室/时间，不展示数据库主键；LLM 可能把数字主题误填成 `reservationId` |
| 我的预约列表应展示本次查询返回的全部有效预约 | 回复文案已经给出总数，截断展示会让用户误以为数据缺失 |
| 创建预约补参不展示手填参会人数 | 参会人数应由参会人选择自动推导；没有参会人时按组织者 1 人兜底，避免前端让用户维护两个可能冲突的字段 |
| 预约详情默认不选择取消/驳回/异常预约 | 用户问“明天下午那个会”时应定位仍有业务意义的预约，历史无效状态不应制造重复选择 |
| 带“概览/概况/统计/摘要”的请求只能接受系统概览工具 plan | LLM 可能把“今天概览怎么样”误解为今日安排；业务查询必须优先走 Tool Registry 中的明确工具 |
| 创建预约设备需求在会议室推荐前单独补参 | 设备会影响会议室筛选，应在基础时间和参会人确定后单独采集，再调用推荐接口 |
| 紧急会议抢占第一版不新增库表字段 | 规格明确限制，抢占结果通过现有 `status/cancel_reason/remark/notification` 表达 |
| 紧急会议确认接口不信任前端预览结果 | 确认时必须重新计算冲突和调配方案，避免预览后预约变化造成双 ACTIVE 占用 |
| 紧急会议确认前使用短期预览指纹 | 不新增抢占关系表，确认时通过 15 分钟内存指纹校验“已预览且占用未变化”；服务重启或超时后要求重新预览 |
| 后端数据库连接通过环境变量注入 | 公开仓库不能包含真实数据库地址或密码；本机启动通过忽略的 `application-local.yml` 沿用当前电脑的 `SERVER_PUBLIC_IP`、`DB_PORT`、`DB_PASSWORD` |
